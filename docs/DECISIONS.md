# Архитектурные решения (ADR)

Короткие записи в формате «контекст → решение → последствия». Объясняют, **почему**
код устроен так, а не иначе — чтобы при доработке не сломать заложенную логику.

---

## ADR-1. Webhook доверяет API ЮKassa, а не телу запроса

**Контекст.** Тело webhook-уведомления приходит по HTTP и легко подделывается. Раньше статус
платежа выставлялся прямо из тела → любой мог пометить платёж оплаченным (SEC-1).

**Решение.** Из тела берём только `id` платежа. Реальный статус/сумму запрашиваем
`GET /v3/payments/{id}` нашими Basic-credentials ([YooKassaClient.getPayment](../server-site/src/main/java/dev/taipan/server_site/yookassa/YooKassaClient.java)).
Доверяем только ответу API. Статический query-token оставлен как дешёвый фильтр от спама.

**Последствия.** Подделать оплату нельзя (нужен реально оплаченный платёж в ЮKassa). На каждый
webhook — один исходящий запрос к API. При сбое проверки возвращаем 502 → ЮKassa повторит.

## ADR-2. Выдача VIP: transactional outbox + RCON + ретраи

**Контекст.** После оплаты VIP надо выдать на игровом сервере. Сервер может быть оффлайн,
RCON — моргнуть, webhook — прийти дважды. Нужна надёжность и идемпотентность.

**Решение.** Двухуровневая схема:
1. В транзакции подтверждения платежа создаётся запись `vip_grants` со статусом `PENDING`
   ([PaymentConfirmationService](../server-site/src/main/java/dev/taipan/server_site/service/PaymentConfirmationService.java)).
   Это «outbox»: durable-факт «надо выдать», уникальный по `payment_id` (не задвоится).
2. Доставка идёт **вне** транзакции ([FulfillmentService](../server-site/src/main/java/dev/taipan/server_site/service/FulfillmentService.java)):
   - **командой** — RCON-команда на сервер (шаблон `rcon.vip-command`, по умолчанию LuckPerms);
   - **другим способом** — если RCON выключен/недоступен, грант остаётся в очереди; его дотягивает
     планировщик ([VipGrantRetryScheduler](../server-site/src/main/java/dev/taipan/server_site/service/VipGrantRetryScheduler.java)),
     и/или игровой плагин может сам читать `vip_grants` и выдавать VIP при входе игрока.

**Контракт очереди** (`vip_grants`): `nick`, `platform`, `duration_days`, `status`, `attempts`.
Статусы: `PENDING → DELIVERED` или `→ FAILED` после `rcon.max-attempts` попыток.

**Последствия.** Оплата никогда не «теряется»: даже при оффлайн-сервере VIP доедет. Сетевой
вызов не держит БД-транзакцию открытой. Идемпотентность — на уровне уникального индекса.

> **⚠ Открытый изъян этого решения (SEC-10 / SITE-9).** Идемпотентно только *создание* гранта,
> но не *доставка*. `deliver()` проверяет статус в памяти без блокировки строки, а вызывается
> из двух мест параллельно (webhook + планировщик). Два потока могут прочитать `PENDING`
> одновременно и выдать VIP дважды. Нужен `SELECT … FOR UPDATE SKIP LOCKED` или атомарный
> `UPDATE … WHERE status='PENDING'`. См. [ADR-11](#adr-11-доставка-гранта-под-блокировкой-строки).

> TODO: плагин-консьюмер `vip_grants` на игровой стороне пока не написан. Он должен читать БД
> сайта, но плагин авторизации ходит в БД бота — это часть архитектурного долга ARCH-4
> (см. [ADR-9](#adr-9-предложено-контракт-интеграции-вместо-общих-таблиц)).

## ADR-3. Секреты — только через переменные окружения

**Контекст.** В репозитории лежали живой bot-token, пароли БД, webhook-секрет, ПДн (SEC-2).

**Решение.** Все секреты читаются из ENV (`${VAR:default}` в Spring, `os.environ` в боте).
В репозитории — только `*.example`-шаблоны с плейсхолдерами и `.gitignore`. Мёртвый `config.yaml`
бота удалён (он не читался кодом и дублировал конфиг).

**Последствия.** Репозиторий безопасно публиковать. Для запуска нужно завести `.env`.
Старый токен/пароли считаются скомпрометированными — подлежат отзыву/смене (ручное действие).

## ADR-4. Один источник конфигурации бота — ENV

**Контекст.** Бот имел два источника: `core/config.py` (ENV) и `config.yaml`, которые расходились.

**Решение.** Единственный источник — ENV через [config.py](../TgBot/src/app/core/config.py).
`config.yaml` удалён. Списки каналов/настройки живут в БД (`required_channels`, `settings`).

**Последствия.** Нет рассинхрона. Все параметры — в одном месте (`.env`).

---

## ADR-5. Доступ к боту — через декоратор `@require_access`

**Контекст.** Каждый хендлер вручную вызывал `if not await guard(...)`. Легко забыть → дыра (BOT-3).

**Решение.** Декоратор [require_access](../TgBot/src/app/bot/middlewares/guard.py) оборачивает
хендлер: сначала `guard` (бан/подписки/профиль), затем сам хендлер. Применён ко всем хендлерам.
Синхронные хелперы (`is_admin`) не декорируются — они вызываются внутри тела.

**Последствия.** Проверку доступа нельзя «забыть» — она в декораторе. Меньше шаблонного кода.

## ADR-6. Таймауты у HTTP-клиента ЮKassa

**Контекст.** `RestClient` без таймаутов: зависший платёжный API заблокировал бы поток запроса.

**Решение.** В [YooKassaClient](../server-site/src/main/java/dev/taipan/server_site/yookassa/YooKassaClient.java)
заданы connect=5с, read=10с.

**Последствия.** Деградация платёжного API не вешает приложение; ошибки ловятся и обрабатываются.

## ADR-7. Меню — один плагин (AstronexisCore)

**Контекст.** `TaipanMenuPlugin` и `AstronexisCore` дублировали меню и оба регистрировали `/menu`.

**Решение.** Вся функциональность меню — в модуле `menu` AstronexisCore (паритет проверён:
страницы, телепорты, компас, cooldown, тексты справки в i18n; плюс Bedrock/action bar).
`TaipanMenuPlugin` удалён.

**Последствия.** Нет конфликта `/menu`, одна точка развития. На сервере должен стоять только
`AstronexisCore.jar`.

## ADR-8. Заморозка неавторизованного игрока — «запрет по умолчанию»

**Контекст.** Гейт 2FA отменял движение/урон/команды, но пропускал чат, блоки, инвентарь (SEC-4).

**Решение.** [GateListener](../TaipanAuthTg/src/main/java/dev/taipan/auth_tg/gate/GateListener.java)
отменяет все значимые действия при `mustBeFrozen`: чат, интеракции, блоки, дроп, инвентарь, подбор.
Плюс лимит попыток `/tgcode` (AUTH-2): после 5 неверных код блокируется, игрок остаётся заморожен.

**Последствия.** До прохождения 2FA игрок реально ничего не может. Перебор кода ограничен.

> **⚠ Решение подорвано реализацией (SEC-8 / AUTH-7).** «Запрет по умолчанию» держится только
> пока в памяти есть запись `pending`. `isBlocked()` при истечении TTL кода **удаляет** запись и
> возвращает `false` → игрок становится авторизованным без ввода кода. То есть «по умолчанию»
> на деле «пока не истёк код», а дальше — «разрешено». Это инверсия принципа. Правильный
> дизайн — позитивный признак авторизации (валидный trust) и fail-closed при истечении, см.
> [ADR-10](#adr-10-предложено-2fa-гейт-как-fail-closed-state-machine).

# Предложенные решения (ADR-9…11) — «как делать дальше»

Записи ниже — не то, что реализовано, а рекомендации по итогам независимого ревью. Они
адресуют системные причины найденных дыр, а не только симптомы.

## ADR-9 (предложено). Контракт интеграции вместо общих таблиц

**Контекст.** `TaipanAuthTg` читает таблицу `bindings` сырым SQL, а её схему определяют
миграции `TgBot`. Сайт пишет `vip_grants` в свою БД, а «плагин-консьюмер» (не написан) должен
читать её же. 4 модуля, 2 базы, связи через прямой доступ к чужим таблицам (ARCH-4).

**Проблема такого подхода.** Нет контракта: изменение схемы одним модулем молча ломает
другой в рантайме, а не на этапе теста/сборки. Две кодовые базы делят приватные таблицы.

**Предложение.** Владелец данных выставляет узкий внутренний API, потребитель ходит в него:
- бот отдаёт `GET /bindings?nick=` (привязки — его домен), плагин дёргает async + кэш;
- доставка VIP идёт через внутренний API игрового кластера **или** общий брокер сообщений,
  а не «плагин почитает таблицу сайта».

**Последствия.** Схема меняется — падает контрактный тест, а не прод. Модули развязаны.
Цена: нужно поднять и версионировать API. Для текущего масштаба — оправданно, т.к. это
устраняет целый класс скрытых поломок.

## ADR-10 (предложено). 2FA-гейт как fail-closed state-machine

**Контекст.** Текущий гейт (ADR-8) держит заморозку на наличии записи `pending` и
размораживает при её отсутствии — отсюда обход по истечению TTL (SEC-8).

**Предложение.** Явные состояния: `AWAITING_LOGIN → AWAITING_CODE → CODE_EXPIRED → TRUSTED`.
Авторизованность определяется **позитивно** (есть валидная trust-сессия ИЛИ код только что
принят), а истечение кода ведёт в `CODE_EXPIRED` (гейт закрыт, нужен повторный `/login`) —
**никогда** в «авторизован». Всю логику вынести в чистый класс без Bukkit → покрыть
unit-тестами (в т.ч. кейс «переждал TTL»). БД-вызовы — только async (устраняет SEC-11).

**Последствия.** Fail-closed by design: любой неучтённый переход оставляет игрока
замороженным, а не пускает. Тестируемо без запущенного сервера. Цена: рефактор GateListener
(пересекается с AUTH-5).

---

## ADR-11 (предложено). Доставка гранта под блокировкой строки

**Контекст.** `FulfillmentService.deliver()` вызывается параллельно из webhook и планировщика
без блокировки — возможна двойная выдача VIP (SEC-10).

**Предложение.** Перед RCON-вызовом атомарно «захватывать» грант:
`UPDATE vip_grants SET status='DELIVERING' WHERE id=? AND status='PENDING'` и продолжать
только если обновлена 1 строка; либо выбирать пачку `SELECT … FOR UPDATE SKIP LOCKED` в
транзакции. RCON-вызов — вне транзакции, финальный статус (`DELIVERED`/`PENDING`+attempts) —
отдельным апдейтом.

**Последствия.** Ровно одна доставка на грант даже при гонке webhook↔планировщик. Идемпотентно
не только создание, но и выдача. Цена: аккуратность с состоянием `DELIVERING` (таймаут на
зависшую доставку, чтобы грант не застрял).
