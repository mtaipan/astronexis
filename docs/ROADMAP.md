# Roadmap: от «не сломано» к «отличному»

Этот документ — про **видение**, а не про точечные фиксы (те живут в [TODO.md](TODO.md)).
Три фазы: сначала заслужить «хорошо», потом «надёжно», потом «отлично».
Не прыгай через Фазу 1 — без неё остальное строится на песке.

Текущее положение — в [STATUS.md](STATUS.md): «не сломано», но не «хорошо».

---

## Фаза 1. Заслужить «хорошо» (фундамент доверия)

Без этого любая дальнейшая работа — вера, а не инженерия.

### 1.1 Тесты там, где есть риск
- **server-site (JUnit + Testcontainers Postgres):**
  - webhook: поддельное тело `status=succeeded` без реального платежа НЕ меняет статус (SEC-1);
  - outbox: повторный webhook не задваивает грант; грант создаётся только для VIP;
  - `FulfillmentService`: успех → DELIVERED; ошибка RCON → attempts++ → FAILED после лимита;
  - `PaymentService`: границы суммы, валюты, нормализация ника;
  - `RconClient`: кодирование/декодирование пакета против фейкового RCON-сервера на сокете.
- **TgBot (pytest + pytest-asyncio):**
  - `BindingsService.bind_strict`: все правила уникальности и запрет перезаписи чужого ника;
  - `AccessService`: бан, подписки, удаление привязки при отписке;
  - `text_router`: маршрутизация по `step`; `require_access` реально блокирует.
- **Плагины (MockBukkit):**
  - `GateListener`: заморожённый игрок не может чат/блоки/инвентарь; лимит попыток кода;
  - `Codes`/`Fingerprints`: формат и стабильность.

### 1.2 Один реальный сквозной прогон (и в runbook)
- Поднять Postgres + сайт + бота локально, прогнать руками:
  - оплата (тестовый магазин ЮKassa) → webhook → грант → RCON-выдача на тестовом сервере;
  - `/bind` → `/login` → код в Telegram → `/tgcode` → разморозка.
- Собрать плагины на **JDK 21** (OPS-4).

### 1.3 Гигиена репозитория
- `git init`, осмысленная история, ветки + PR.
- **CI (GitHub Actions):** matrix — Java (JDK 21, gradle+maven) и Python (lint+test) на каждый PR.
- Линтеры/форматтеры: `ruff`+`black` (Python), Spotless/Checkstyle (Java), `pre-commit`.
- Зависимости: pin + Dependabot/renovate; проверить происхождение jar в `libs/`.

**Критерий выхода из фазы:** CI зелёный, ключевые пути под тестами, всё хоть раз отработало вживую.

## Фаза 2. Сделать надёжным (production-grade)

### 2.1 Деньги без потерь
- **Реконсиляция:** периодический джоб опрашивает ЮKassa по «зависшим» PENDING-платежам
  (не полагаться только на webhook — он может не дойти).
- **Dead-letter + алерт:** грант в FAILED → уведомление админу (бот) + видно в отчёте.
- Идемпотентность и аудит на каждом денежном шаге (уже частично: outbox, idempotence-key).

### 2.2 Наблюдаемость
- Структурные логи (JSON) + **correlation id** (`payment_id` через весь путь: сайт → грант → RCON).
- Метрики через Spring Boot Actuator + Micrometer/Prometheus: счётчики платежей, грантов,
  ошибок webhook, времени доставки. Дашборд «оплачено, но не выдано».
- Health-checks для всех сервисов; алертинг на FAILED-гранты и 5xx webhook.

### 2.3 Чистая интеграция «сайт ↔ игра»
Сейчас доставка завязана на RCON-из-веба + неявный контракт по таблице. Лучше:
- Вынести **выдачу прав за API**: небольшой эндпоинт «выдать/проверить entitlement» с авторизацией
  (API-ключ/mTLS), который дёргает сайт, а применяет игровой плагин. Это убирает RCON-из-веба
  и кросс-БД-хак, делает контракт явным и тестируемым.
- Альтернатива полегче: плагин-консьюмер читает `vip_grants` по защищённому каналу.

### 2.4 Безопасность по умолчанию
- Rate-limiting на формах платежей и на боте (bucket4j / собственный лимитер) — SITE-4.
- Spring Security + CSRF где есть состояние; least-privilege пользователи БД.
- RCON только во внутренней сети; ротация секретов; секрет-менеджер (Vault/SOPS/Doppler)
  вместо plain `.env` в проде.
- Аудит-лог админских действий (бан, смена привязки, выдача).
- Авторизация: события AuthMe вместо поллинга (AUTH-3); фолбэк доставки кода при сбое TG (AUTH-6).

**Критерий выхода:** платёж невозможно «потерять», инциденты видно в метриках/алертах,
интеграция с игрой — через явный контракт, а не «дёрнем RCON и понадеемся».

## Фаза 3. Сделать отличным (то, чем гордятся)

### 3.1 Архитектура: единый «платформенный» контекст
- Сейчас три почти-независимых куска с пересечением по сущностям (ник, привязка, entitlement)
  и **двумя БД** (сайт `server_site`, бот/авторизация `tgbot`). Это источник трения.
- Отличный вариант: выделить **единый сервис идентичности/прав** (bindings + entitlements),
  с которым по API общаются сайт, бот и плагины. Один источник правды, явные границы,
  лёгкая замена любого клиента.
- Событийная модель: платёж публикует `VipGranted` → подписчики (выдача, уведомление, статистика).

### 3.2 Переиспользование между плагинами
- `TaipanAuthTg` и `AstronexisCore` дублируют инфраструктуру (Telegram API, конфиг, определение
  платформы, i18n). Вынести в общую библиотеку `taipan-commons`. Меньше копипасты, единый стиль.

### 3.3 Доменное моделирование
- Деньги — тип `Money(amount, currency)`, а не голый `BigDecimal` по всему коду.
- Value-objects для ника/платформы; явная таксономия ошибок вместо `IllegalArgumentException` везде.

### 3.4 Продукт и UX
- Самообслуживание: игрок видит статус VIP, историю платежей, может перезапросить выдачу.
- Веб-админка вместо только Telegram-команд; аудит и поиск.
- i18n сайта; доступность; аккуратные страницы статусов оплаты.

### 3.5 Поставка и эксплуатация
- Единый воспроизводимый деплой (один compose / k8s), staging-окружение, zero-downtime деплой.
- Бэкапы Postgres + регулярные **учебные восстановления** (бэкап без проверки восстановления — не бэкап).
- Runbooks: «платёж не доставлен», «бот лёг», «ротация токена», «откат миграции».
- SLO/мониторинг доступности; отчёты.

**Критерий «отлично»:** новый разработчик за день поднимает проект и безопасно катит изменение;
платёж и выдача наблюдаемы и невозможны к потере; модули независимы и переиспользуемы;
инциденты разбираются по runbook, а не по наитию.

## Принцип, который отличает «отлично» от «работает»

> Хорошие проекты получаются не из отсутствия видимых багов, а из того, что **каждое важное
> поведение проверяемо** (тестом, метрикой, алертом) и **каждый сбой предусмотрен** (ретрай,
> dead-letter, runbook). Сейчас у проекта правильный скелет — не хватает именно этой нервной
> системы.
